<span data-ttu-id="e9804-104">Wie alle [*Prozesse*](/windows/desktop/SecGloss/p-gly)verfügt ein geschützter Server über ein [*Primäres Zugriffs Token*](/windows/desktop/SecGloss/p-gly) , das seinen [*Sicherheitskontext*](/windows/desktop/SecGloss/s-gly)beschreibt.</span><span class="sxs-lookup"><span data-stu-id="e9804-104">Like all [*processes*](/windows/desktop/SecGloss/p-gly), a protected server has a [*primary access token*](/windows/desktop/SecGloss/p-gly) that describes its [*security context*](/windows/desktop/SecGloss/s-gly).</span></span> <span data-ttu-id="e9804-105">Wenn ein Client eine Verbindung mit einem geschützten Server herstellt, möchte der Server möglicherweise Aktionen mithilfe des Sicherheits Kontexts des Clients anstelle des Sicherheits Kontexts des Servers ausführen.</span><span class="sxs-lookup"><span data-stu-id="e9804-105">When a client connects to a protected server, the server may want to perform actions using the client's security context instead of the server's security context.</span></span> <span data-ttu-id="e9804-106">Wenn z. b. ein Client in einer DDE-Konversation (Dynamic Data Exchange) Informationen von einem DDE-Server anfordert, muss der Server überprüfen, ob der Client Zugriff auf die Informationen hat.</span><span class="sxs-lookup"><span data-stu-id="e9804-106">For example, when a client in a dynamic data exchange (DDE) conversation requests information from a DDE server, the server needs to verify that the client is allowed access to the information.</span></span>

<span data-ttu-id="e9804-107">Es gibt zwei Möglichkeiten, wie ein Server im Sicherheitskontext des Clients agieren kann:</span><span class="sxs-lookup"><span data-stu-id="e9804-107">There are two ways that a server can act in the client's security context:</span></span>

-   <span data-ttu-id="e9804-108">Ein Thread des Server Prozesses kann die Identität des Clients annehmen.</span><span class="sxs-lookup"><span data-stu-id="e9804-108">A thread of the server process can impersonate the client.</span></span> <span data-ttu-id="e9804-109">In diesem Fall verfügt der Thread des Servers über ein Identitätswechsel- [*Zugriffs Token*](/windows/desktop/SecGloss/a-gly) , das den Client, die Gruppen des Clients und die [*Berechtigungen*](/windows/desktop/SecGloss/p-gly)des Clients identifiziert.</span><span class="sxs-lookup"><span data-stu-id="e9804-109">In this case, the server's thread has an impersonation [*access token*](/windows/desktop/SecGloss/a-gly) that identifies the client, the client's groups, and the client's [*privileges*](/windows/desktop/SecGloss/p-gly).</span></span> <span data-ttu-id="e9804-110">Weitere Informationen finden Sie unter [Client](client-impersonation.md)Identitätswechsel.</span><span class="sxs-lookup"><span data-stu-id="e9804-110">For more information, see [Client Impersonation](client-impersonation.md).</span></span>

<span data-ttu-id="e9804-115">In den meisten Fällen ist es ausreichend, die Identität des Clients anzunehmen.</span><span class="sxs-lookup"><span data-stu-id="e9804-115">In most cases, impersonating the client is sufficient.</span></span> <span data-ttu-id="e9804-116">Der Identitätswechsel ermöglicht es dem Server, den Client Zugriff auf Sicherungs fähige Objekte zu überprüfen, die Berechtigungen des Clients zu überprüfen und Überwachungs Pfad Einträge zu generieren, die den Client identifizieren.</span><span class="sxs-lookup"><span data-stu-id="e9804-116">Impersonation enables the server to check the client's access to securable objects, check the client's privileges, and generate audit trail entries that identify the client.</span></span> <span data-ttu-id="e9804-117">In der Regel muss ein Server nur dann eine Client Anmelde Sitzung starten, wenn er den Sicherheitskontext des Clients für den Zugriff auf Netzwerkressourcen verwenden muss.</span><span class="sxs-lookup"><span data-stu-id="e9804-117">Typically, a server needs to start a client logon session only if it needs to use the client's security context to access network resources.</span></span>
