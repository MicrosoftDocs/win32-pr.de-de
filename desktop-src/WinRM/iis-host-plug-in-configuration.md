# <a name="iis-host-plug-in-configuration"></a>IIS-Host-Plug-in-Konfiguration

Die Konfiguration des Host-Plug-Ins für Windows-Remoteverwaltung (WinRM) Internetinformationsdienste (IIS) bietet einen Hostingmechanismus, mit dem WinRM-Plug-ins von Drittanbietern Verwaltungsdaten innerhalb der WinRM-Infrastruktur verfügbar machen können.

In den folgenden Abschnitten wird die Konfiguration des IIS-Plug-Ins beschrieben:

-   [IIS-hostingdetails](#iis-hosting-details)
-   [IIS-Host Konfigurations Modell](#iis-host-configuration-model)
-   [Setup und Details zum IIS-Host (WinRM)-Konfiguration](/windows)
-   [IIS-Host Konfiguration wird gelesen](#reading-iis-host-configuration)
-   [Auflisten von Plug-Ins für den WinRM-Dienst](#enumerating-plug-ins-on-the-winrm-service)

## <a name="iis-hosting-details"></a>IIS-hostingdetails

Eine Anwendung, die den WinRM-SOAP-Prozessor in IIS konfiguriert und initialisiert, empfängt Anforderungen direkt von einem WinRM-Client auf einem anderen Computer mithilfe des WS-Management Protokolls. Die Anwendung muss außerdem einen HTTP-Endpunkt konfigurieren, der die Anwendung identifiziert, um sicherzustellen, dass alle Anforderungen direkt an die anwendungsspezifischen Plug-Ins weitergeleitet werden. Die Anwendung muss außerdem konfigurieren, welche Plug-ins in den IIS-Host geladen werden sollen und welche Autorisierungs-und Authentifizierungsmechanismen benötigt werden. Wenn Sie den WinRM-Dienst in IIS verwenden, sollten die folgenden Faktoren berücksichtigt werden:

-   Die Kerberos-Authentifizierung wird von IIS standardmäßig nicht unterstützt. Es kann jedoch ein IIS-Modul erstellt werden, um die Kerberos-Authentifizierung zu unterstützen.
-   Der IIS-Dienst akzeptiert keine verschlüsselten WinRM-Dienst Anforderungen über http.
-   Nachdem Sie einen IIS-Endpunkt eingerichtet haben, tritt bei der ersten Anforderung, die Sie mit impliziten Anmelde Informationen senden, ein Fehler auf. alle nachfolgenden Anforderungen können jedoch erfolgreich ausgeführt werden. Dies ist ein bekanntes Problem bei einer IIS-Site, die mit der Standardkernel Modus-Authentifizierung konfiguriert ist. Wenn dies ein Blockierungs Problem ist, empfiehlt es sich, die Kernel Modus-Authentifizierung zu deaktivieren.

## <a name="iis-host-configuration-model"></a>IIS-Host Konfigurations Modell

Alle Anwendungen, die einen WinRM-SOAP-Prozessor (Simple Object Access Protocol) in IIS hosten, müssen konfiguriert werden, bevor Sie funktionieren.

Jede Anwendungskonfiguration ist von anderen Anwendungen getrennt. Die folgenden Konfigurationen sind erforderlich, obwohl möglicherweise nicht alle Konfigurationen geändert werden müssen, und Sie werden über den WinRM-Hauptdienst durchgeführt:

-   Allgemeine Anwendungskonfiguration, einschließlich Vorgangs Timeouts.
-   Ein Autorisierungs-Plug-in zum Autorisieren von LiveID-Anforderungen oder zum Mapping des Windows-Benutzerkontos zu einem allgemeineren Dienst Konto.
-   Vorgangs-Plug-ins, die gehostet und vom WinRM-SOAP-Prozessor verwendet werden, um die eigentlichen Anforderungen auszuführen.

Die folgenden Konfigurationen müssen möglicherweise in IIS konfiguriert werden:

-   [](windows-remote-management-glossary.md) Listenerendpunktinformationen, die definieren, welche Ports und Adressen mit allen Secure Sockets Layer (SSL)-Zertifikaten auf Anforderungen lauschen.
-   Client Zertifikat Zuordnung zum Zuordnen von Client Zertifikaten zu Benutzerkonten.
-   Autorisierungs Konfiguration, um einzuschränken, wer Zugriff auf die virtuelle Site hat.

Der WinRM-Dienst registriert Port 5985 standardmäßig mit dem Präfix "wsman". Wenn der Kompatibilitäts-http-Listener aktiviert ist, registriert der WinRM-Dienst standardmäßig Port 80 mit dem Präfix "wsman". Da IIS auch standardmäßig an Port 80 lauscht, wird empfohlen, dass die IIS-Host Dienste Listener an einem anderen [*Port erstellen.*](windows-remote-management-glossary.md)

Wenn sowohl der IIS-als auch der WinRM-Dienst Port 80 verwenden, ist die folgende Konfiguration erforderlich, um den Port freizugeben:

-   Der IIS-Host Dienst muss WinRM für eine Anwendung unter dem-Standort aktivieren.
-   Der Anwendungsname muss dem URI hinzugefügt werden. Beispiel: https://servername/myapplication.

Jede IIS-Host Anwendung wird in der IIS-Konfiguration als web.config Datei gespeichert, die sich im physischen Verzeichnis befindet, das dem konfigurierten virtuellen Verzeichnis entspricht. Im folgenden Abschnitt wird beschrieben, wie das IIS-Konfigurationssystem funktioniert und wie es erweitert werden kann, um die WinRM-IIS-Host Konfiguration zu unterstützen.

### <a name="how-iis-configuration-works"></a>Funktionsweise der IIS-Konfiguration

Die IIS-Konfigurations Infrastruktur besteht aus einem Satz von XML-Dateien zum Speichern von Konfigurationsdaten und einem Satz von Schema Dateien, die zum Überprüfen von Daten verwendet werden, die in IIS-Verwaltungs-APIs geschrieben oder gelesen wurden. Auf der Stamm Ebene befindet sich eine ApplicationHost.config-Datei im gleichen Verzeichnis wie IIS (in der Regel% windir% \\ system32 \\ inetsrv \\ config). Dies ist die Hauptkonfigurationsdatei für IIS, bei der die Liste der Websites, Anwendungen, virtuellen Verzeichnisse, allgemeinen Einstellungen, Protokollierung, Zwischenspeicherung usw. gespeichert wird. Konfigurationsdaten werden in als unabhängige Entitäten, die als **Abschnitte** bezeichnet werden, unterteilt.

Beispielsweise gibt es separate **Abschnitte** für Websites, Protokollierung, Caching usw. Alle in ApplicationHost.config enthaltenen Konfigurationsdaten werden auf alle in IIS konfigurierten Standorte, Anwendungen und virtuellen Verzeichnisse angewendet. Auf der Ebene des virtuellen Verzeichnisses definiert eine web.config Datei anwendungsspezifische Konfigurationsdaten, möglicherweise durch Überschreiben anderer Daten, die bereits in ApplicationHost.config definiert sind. Web.config Dateien werden im Stammverzeichnis (physisches Verzeichnis) gespeichert, das der Website, der Anwendung oder dem virtuellen Verzeichnis entspricht, für die die Konfiguration gilt.

Weitere Informationen zur IIS-Konfiguration finden Sie unter:

-   [IIS 7,0-Einstellungs Schema](/previous-versions/iis/settings-schema/aa347559(v=vs.90))
-   [/archive/blogs/carlosag/](/archive/blogs/carlosag/)
-   [https://www.iis.net/articles/view.aspx/IIS7/Use-IIS7-Administration-Tools/Using-XML-Configuration/Deep-Dive-into-IIS7](https://www.iis.net/learn/get-started/planning-your-iis-architecture/deep-dive-into-iis-configuration-with-iis-7-and-iis-8)

Das IIS-Konfigurations Schema zum Überprüfen von IIS-Konfigurationsdaten sind XML-Dateien, die im IIS-Schema Verzeichnis gespeichert werden (in der Regel% windir% \\ system32 \\ inetsrv \\ config \\ Schema). Das IIS-Konfigurations Schema ist nicht XSD. Die IIS-Konfiguration unterstützt die Erweiterbarkeit. Neue Konfigurations Abschnitte können direkt in das IIS-Konfigurationssystem integriert werden, und die Einstellungen des IIS-Konfigurations Systems können mithilfe der vorhandenen Verwaltungs-APIs bearbeitet werden.

Die folgenden Hauptschritte sind erforderlich:

-   Definieren Sie einen neuen Konfigurations Abschnitt in einer XML-Schema Datei.
-   Platzieren Sie die Datei im IIS 7-Schema Verzeichnis (% windir% \\ system32 \\ inetsrv \\ config \\ Schema).
-   Registrieren Sie den neuen Abschnitt in der globalen IIS-Konfigurationsdatei (ApplicationHost.config).

## <a name="iis-host-winrm-configuration-setup-and-details"></a>Setup und Details zum IIS-Host (WinRM)-Konfiguration

In diesem Abschnitt werden die Schritte beschrieben, die während des Setups zum integrieren und Initialisieren der IIS-Host Konfiguration durchgeführt werden müssen. Hier bezieht sich die IIS-Host Konfiguration auf die Konfigurationsdaten, die zum Ausführen des WinRM-Diensts innerhalb von IIS erforderlich sind. Das WinRM IIS-Erweiterungsmodul ist eine optionale Komponente, die mithilfe des- **Server-Manager** installiert wird.

Der IIS-Konfigurations Abschnitt, der die erforderliche Konfiguration definiert, wird als **system.management.wsmanagement.config** bezeichnet. In der Regel wird die IIS-Host Konfiguration während des Setups definiert. Hierzu können Sie eine web.config-Datei erstellen und den entsprechenden XML-Code hinzufügen. Alle Änderungen an der web.config Datei erfordern einen Neustart der Website, auf der der WinRM-Dienst gehostet wird.

Im folgenden Verfahren wird das Einrichten der WinRM-IIS-Erweiterung ausführlich beschrieben.

**So installieren Sie die WinRM-IIS-Erweiterung**

1.  Klicken Sie im Abschnitt " **featurezusammenfassung** " des **Server-Manager** auf **Features hinzufügen**.
2.  Wählen Sie im Assistenten zum Hinzufügen von Features die Option WinRM IIS-Erweiterung aus.
3.  Befolgen Sie die Anweisungen des Assistenten, einschließlich der Installation der erforderlichen Software, falls vorhanden. Weitere Informationen zur Verwendung des Assistenten zum Hinzufügen von Features finden Sie in der **Server-Manager** -Online Hilfe.

### <a name="initializing-winrm-configuration-for-the-hosted-application"></a>WinRM-Konfiguration für die gehostete Anwendung wird initialisiert.

Dieser Schritt umfasst das Definieren der IIS-hostkonfigurations-XML in der web.config-Datei. Diese Datei befindet sich im physischen Verzeichnis, das dem virtuellen Verzeichnis entspricht, in dem die gehostete Anwendung konfiguriert ist. Der XML-Inhalt für den **system.management.wsmanagement.config** Abschnitt sollte dem Schema entsprechen, das in der "wsmanconfig- \_schema.xml definiert ist. andernfalls können die IIS-Verwaltungs-APIs es nicht verarbeiten. Im folgenden finden Sie ein Beispiel für XML-Inhalt, der in der web.config-Datei enthalten sein könnte:


```XML
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
 <system.webServer>
  <system.management.wsmanagement.config>
   <OperationsConfiguration MaxEnvelopeSizeKB="150" 
                            MaxOperationTimeoutSeconds="60"  
                            MaxEnumerateIdleTimeoutSeconds="60"  
                            MaxSubscribeIdleTimeoutSeconds="60"  
                            MaxShellIdleTimeoutMinutes="15"
                            QuotaRefreshIntervalSeconds="900"
                            QuotaSystemWindowSeconds="2"
                            QuotaSystemWindowOperations-"1000"/>
    <PluginModules>
     <AuthorizationPlugin Name="Auth1" Filename="Auth.dll" SDKVersion="1" SupportsQuota="true"/>
     <OperationsPlugins>
      <Plugin Name="PL1" Filename="PL1.dll" SDKVersion="1" XmlRenderingType="text">
       <InitializationParameters>
        <Param name="myParam1" value="myValue1"/>
       </InitializationParameters>
       <Resources>
        <Resource ResourceUri="myUri1">
        <Capability Type="Shell"/>
        </Resource>
       </Resources>
      </Plugin>
      <Plugin Name="PL2" Filename="PL2.dll" SDKVersion="1" XmlRenderingType="text">
       <InitializationParameters/>
       <Resources>
        <Resource ResourceUri="myUri1">
         <Capability Type="Get"/>
         <Capability Type="Put"/>
         <Capability Type="Delete"/>
        </Resource>
       </Resources>
      </Plugin>
     </OperationsPlugins>
    </PluginModules>
   </OperationsConfiguration>
  </system.management.wsmanagement.config>
 </system.webServer>
</configuration>
```

## <a name="reading-iis-host-configuration"></a>IIS-Host Konfiguration wird gelesen

Die IIS-Host Konfiguration wird während des Start Prozesses des SOAP-Prozessors in der gehosteten Anwendung gelesen. Da web.config Dateien manuell bearbeitet werden können und daher anfällig für Schema Validierungs Fehler sind, kann es beim Lesen einer Konfiguration zu Problemen kommen. Während eines Konfigurations Lese-oder Validierungs Fehlers kann die gehostete Anwendung nicht gestartet werden und gibt eine entsprechende Fehlermeldung an den Client zurück. Es wird ein Ereignis protokolliert, das eine ausführliche Beschreibung der Ursache des Fehlers enthält.

Häufige Fehler sind u. a. folgende:

-   Zugriff verweigert
-   Ungültige Konfigurations-XML
-   Es ist kein Plug-in konfiguriert (mindestens eine ist erforderlich)

Die Standard-Plug-ins sind nicht standardmäßig konfiguriert. Sie müssen explizit in der web.config-Datei konfiguriert werden, bevor Sie mit dem empfangen von Anforderungen beginnen.
