# <a name="graphics-low-level-client-support"></a>Grafik Low-Level Clientunterstützung

\[Diese Funktionen können bei jeder Betriebssystemrevision geändert werden. Verwenden Sie stattdessen Microsoft DirectDraw und Microsoft Direct3DAPIs. diese APIs isolieren Anwendungen vor solchen Betriebssystemänderungen und verbergen viele andere Schwierigkeiten bei der direkten Interaktion mit Anzeigetreibern. Weitere Informationen finden Sie unter [Einführung in die Anzeige von](/windows-hardware/drivers/display/introduction-to-display--windows-2000-model-).\]

DirectDraw und Direct3D verwenden einige Kernelmodusroutinen, um mit dem Betriebssystem und dem Anzeigetreiber zu kommunizieren.

Der Betriebssystemkernel scheint aus Sicht der Benutzermoduskomponenten ein DirectDraw/Direct3D-Anzeigetreiber zu sein. Es gibt jedoch einige Unterschiede zwischen Kernel- und Anzeigetreibern. Die Kernelmoduskomponenten behalten eine Kopie von Strukturen bei, die DirectDraw-/Direct3D-Objekte darstellen, z. B. Geräte und Oberflächen. Die Benutzermoduskomponente von DirectDraw/Direct3D fordert die Erstellung dieser Strukturen an, auf die mit Handles verwiesen wird, die vom Kernelmodus zurückgegeben werden. Folglich liegen die Unterschiede hauptsächlich in den Argumenten, die an die Routinen übergeben werden: Kernel akzeptiert Handles, während Treiber in der Regel Zeiger akzeptieren.

## <a name="evolution-of-kernel-mode-entry-points"></a>Entwicklung von Einstiegspunkten im Kernelmodus

DirectDraw stellt alle Verbindungen mit dem Kernelmodus über die Gdi32.dll mit mehreren Einstiegspunkten her. Diese Einstiegspunkte marshallen in der Regel nur Parameter. Im Rahmen der Geräteerstellung gibt der Gdi32.dll Zeiger auf Routinen in sich selbst zurück, die dann von DirectDraw und Direct3D 7.0 verwendet werden.

Direct3D 8. *x* unterbricht Gdi32.dll und ruft die Einstiegspunkte im Kernelmodus direkt auf.

## <a name="calling-the-kernel"></a>Aufrufen des Kernels

Diese Einstiegspunkte sind ähnlich und in vielen Fällen identisch mit Einstiegspunkten im Anzeigetreiber selbst. Daher ist ein Verständnis der DDK-Materialien (Driver Development Kit) für DirectDraw und Direct3D eine wesentliche Voraussetzung für die Verwendung dieser Funktionen.

Das Aufrufen dieser Funktionen unterscheidet sich von Betriebssystem zu Betriebssystem. Am Windows 2000 der W2KUMODE. Die LIB-Bibliothek enthält Routinen, die es dem aufrufenden Thread ermöglichen, den Übergang in den Kernelmodus vorzunehmen. Diese Routinen verwenden kein Parameter marshallen und sind der ideale Aufrufmechanismus für Windows 2000. Der Mechanismus, der den Übergang in den Kernelmodus vornimmt, basiert auf der Reihenfolge einer Tabelle. Diese Reihenfolge ändert sich über Betriebssystemrevisionen hinweg. Daher ist es nicht ratsam, eine Anwendung zu erstellen, die auf W2KUMODE basiert. LIB für Windows 2000 und erwarten, dass es auf Windows XP unverändert ausgeführt wird. Ein Mechanismus zum Erstellen der Portabilität des Betriebssystems wird im folgenden Abschnitt beschrieben.

Auf Windows XP exportiert D3D8THK.DLL alle Kernelmodusfunktionen, jedoch mit einem leicht ergänzten Namen. Das folgende Beispiel zeigt, wie Anwendungen die Clientfunktionen auf niedriger Ebene mithilfe des Präprozessors aliasieren können. Verknüpfen Sie zuvor dynamisch mit D3D8THK.DLL.


```

#define NtGdiDdUnlock OsThunkDdUnlock
#include "ntgdi.h"      // Brings in the definition of "NtGdiDdUnlock", 
.                       //   which is now seen as OsThunkDdUnlock.

```



## <a name="writing-portable-applications"></a>Schreiben portabler Anwendungen

Es stehen mehrere Verfahren zur Verfügung, um eine Anwendung betriebssystemübergreifend portierbar zu machen. Am stabilsten ist die Installation des verteilbaren Microsoft DirectX 8.x-Pakets auf Windows 2000 im Rahmen des Installationsprozesses Ihrer eigenen Anwendung. Dadurch wird sichergestellt, dass D3D8THK.DLL auf Windows 2000-Systemen verfügbar ist und einen einzelnen Codepfad in der Anwendung zulässt, da Sie das gleiche Verfahren wie oben für Windows XP verwenden können.

Anwendungen können auch eine eigene isolierende DLL erstellen und zwei Versionen dieser DLL implementieren: eine für Windows 2000 mithilfe der oben gezeigten Technik und eine für Windows XP, die D3D8THK.DLL verwendet.

## <a name="device-creation"></a>Geräteerstellung

DirectDraw und Direct3D müssen zunächst eine Instanz der Kernelabstraktion des Treiberobjekts erstellen. In Gdi32.dll und Ddraw.dll wird dies durch Aufrufen von [**DdCreateDirectDrawObject,**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatedirectdrawobject) [**DdQueryDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddquerydirectdrawobject) und [**DdReenableDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddreenabledirectdrawobject)erreicht. In Direct3D 8. *x*, dies wird erreicht, indem [**NtGdiDdCreateDirectDrawObject,**](-dxgkernel-ntgdiddcreatedirectdrawobject.md) [**NtGdiDdQueryDirectDrawObject**](-dxgkernel-ntgdiddquerydirectdrawobject.md) und [**NtGdiDdReenableDirectDrawObject**](-dxgkernel-ntgdiddreenabledirectdrawobject.md)aufgerufen werden. Beachten Sie, dass die \* GdiEntry-Einstiegspunkte lediglich Marshaller für die Einstiegspunkte im Kernelmodus sind.

Nach dem Aufrufen dieser Funktionen wird ein Handle zurückgegeben, das die Abstraktion des Kernelobjekts eines DirectDraw-Geräts darstellt. Dieses Kernelobjekt stellt eine Treiberinstanz dar, die im Besitz des Anzeigetreibers ist. Dieses Handle kann in nachfolgenden Aufrufen wie [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (oder [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) in Direct3D 8) verwendet werden.*x*), um weitere Objekte zu erstellen.

## <a name="kernel-objects"></a>Kernelobjekte

Einige Einstiegspunkte werden verwendet, um die Kernelmodusdarstellungen bestimmter Objekte zu verwalten. [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (oder [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) in Direct3D 8.*x*) ist ein gutes Beispiel. Dieser Einstiegspunkt erstellt ein Kernelmodusobjekt, das einen Teil des Systemspeichers darstellt. Der Einstiegspunkt gibt ein Handle zurück, und dieses Handle kann in nachfolgenden Zeichnungsaufrufen verwendet werden. Der Kernel leitet einen Zeiger auf seine eigene Struktur von diesen Handles ab und übergibt diesen Zeiger an den Anzeigetreiber, der dann den Vorgang ausführt.
